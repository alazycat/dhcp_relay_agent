Implementation Notes

Issue #1 — run() refactor: spawn_v4_tasks() / spawn_v6_tasks() — 2026-06-15

Design Decisions

Decision Private helpers with #[cfg] instead of trait-based handlers

Ambiguity: SPEC §4.2 proposed internal helpers; could also use trait-based RelayHandler pattern.

Choice: #[cfg(feature)] private methods on RelayAgent — spawn_v4_tasks() and spawn_v6_tasks().

Rationale: Trait-based handler adds ~30 lines of abstraction with no immediate value. cfg-gated methods are 3 dispatch lines in run(). Trait approach deferred to v0.3 relay-core.

Decision v6 stub returns empty instead of blocking

Choice: spawn_v6_tasks() subscribes to shutdown (validates channel) but returns Ok(Vec::new()) immediately.

Rationale: Blocking in a stub would prevent v4 from running when dhcpv6 is accidentally enabled. Empty return is safe and lets issue #3 drop in without changes to run().

Deviations

Deviation Removed dual-#[cfg] run() pattern

Old code: Two run() methods gated with #[cfg(feature = "dhcpv4")] / #[cfg(not(...))].

New code: Single unified run() with internal #[cfg] blocks. When no protocol features enabled, returns immediately.

Why: Dual signatures with combinational cfg gates don't scale. Internal dispatch adds one cfg block per protocol — linear scaling.

Tradeoffs

Tradeoff Immediate return vs blocking when no features enabled

Alternatives: (A) Subscribe to shutdown and block; (B) Return immediately.

Chose B: Zero tasks = nothing to do. Blocking would be confusing behavior for callers.

Downside: Callers expecting unconditional block must handle immediate return. Mitigated: dhcpv4 is the default feature; edge case only triggers with explicit default-features = false.

Open Questions

Question v6 stub requires dhcpv6 feature + dhcproto/v6?

The spawn_v6_tasks stub is gated behind #[cfg(feature = "dhcpv6")] which activates dhcp/v6 module. Verify cargo build --features dhcpv6 compiles correctly when dhcproto v6 feature is aligned — current Cargo.toml does not propagate v6 to dhcproto.
